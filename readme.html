1. Installation

Prinzipielle ist das ganze Projekt bisher noch nicht in einem finalen deploybaren Zustand. Ich hatte einfach bisher keine Zeit mich damit auseinanderzusetzen. Aber alle Funktionen können zumindest in den Testumgebungen von Ionic und Spring Boot verwendet werden.

Inez-Ionic

Ist eine PWA. Warum? Siehe IONIC-PWA

Um die PWA zu deployen wird NPM und Node.js benötigt: Auf https://nodejs.org/en/ node.js runterladen und installieren.

Dann wird noch Ionic benötigt: dafür in der Konsole npm install -g ionic

Um die APP zustarten:

In der Konsole zum Root-Ordner (inez-ionic) navigieren und dann ionic serve eingeben. Zuvor sollte man npm update machen um sie die Packages reinzuladen (im Root-Ordner). Alternativ kann man ionic serve --devapp eingeben und die APP Ionic Dev App runterladen. In ihr kann man die APP direkt auf dem Smartphone testen.

Inez-Web

Variante 1: Projekt in Eclipse (oder andere IDE, aber ohne Gewähr) importieren als Maven Projekt und de.david.inez.Application als Java Application ausführen

2. Allgemeine Architektur

Das Programm teilt sich in zwei Parts auf. Einmal die Backend mit verschiedenen Services und Datenspeicherung, die praktisch auf ein einem Server laufen sollte. Zum anderen gibt es die Frontend, die mit Ionic geschrieben wurde. Also eine PWA. Als APP und Webapp nutzbar.


FRONTEND BACKEND
IONIC-PWA
App als auch Webapp
Die Nutzung als APP ist vorgesehen und wurde bevorzugt in der Entwicklung. Zwar sind native Apps verboten, aber da es sich hierbei um eine progressive Web App handelt und somit plattformunabhänig ist (egal ob Android, IOS oder ein Browser) habe ich mich trotzdem für diesen Fokus entschieden, da nach meiner Ansicht ein Einkaufslisten-Service nur als APP wirklich Verwendung findet und sinnvoll ist.
In der App kann die IP-Adresse des Backend-Servers eingestellt werden. Wenn das Projekt auf einem Rechner läuft, kann man einfach localhost:8080 verwend (Port ist immer 8080).
Basierend auf Spring
Spring Boot um das Projekt schneller zu starten. Produtiv nicht sinnvoll, aber für diesen Zweck durchaus angemessen.
Rest-Webservice als Schnittstelle zur Frontend

5. Autocomplete und Co

Das Programm besitzt ein Vorschlagsystem, das zu einem Input Vorschläge macht und dabei auf Rating setzt.

5.1 Vorteile des Vorschlagsystems:

  • Erkennung von verschiedenen Objekte aus der Datenbank im Inputstring. Z.B. Produkte, Einheiten, Numeral etc.
  • Tippfehler werden durch das Rating ausgemerzt.

5.2 Aufbau der Vorschlagsystems

Das System nutzt den SimilairtyService. Dieser erstellt für eine Satz an Daten ein Rating für den mitgegeben Inputstring. Näheres später.

Das System kann unterschiedliche Klassen erkennen und dadurch die eingegben Werte anpassen. Z.B. andere Namen für die Einheit ausgeben oder an das Einheitensystem anpassen sowie Produktnamen korrigieren und Preise ausgeben. Da ein Produkt einen Name besitzt der über die Werte Singular & Plural verfügt wird der Produktname je nach Anzahl angepasst. Das gilt auch für Einheiten (Unit). Z.B. ein Kasten oder zwei Kästen.

5.3 Der SimilarityService

Dieser Service ist eine Kernfunktion von meiner INEZ-Variante. Ausgehenden von Datensätzen können diese Datensätze nach Ähnlichkeiten in den String bewertet werden.

Es gibt zwei Versionen:

  • SimilarityService
  • ComplextSimilartiyService

5.4 SimilarityService

Einfache Version. Gibt ein einfaches Rating ab für eine compareableValue und dem Input. Baut auf dem ComplexSimilarityService auf.

5.5 ComplexSimilarityVersion

Dieser Service macht prinzipiell das gleiche wie der SimilarityService mit nur einem Unterschied. Er bezieht neben dem Standardrating noch weiter externe Daten mit ein. Beispielsweise soll die das Rating für verschiede Units erstellt werden. Aber der Inputstring bestitz auch ein Produkt. Nur werden die Produkte aus der Datenbank als externe Daten verwedent um die einzelnen Buchstaben des Inputstirngs zu bewerten. Also wie sehr sie von den exteren Daten verwendet werden. So sollte das erste "l" in 1l Milch für eine Unit wesentlich wichtiger sein, als das in "Milch". Das vorgezogene InputRating sorgt dafür (InputRating wird nur bei Units gemacht).

5.6 Rating

Ratingkriterien

  • Anzahl der gleichen Buchstaben
  • Gleiche Buchstabensequenzen
  • Längste Buchstatensequenz
  • Vergleich der Länge der Texte

5.7 NumberService

Der NumberService erkennt Zahlen und Zahlworte und kann diese sogar durch Mutliplikation verbinden. Z.B: 2 Paar... -> 4

Die Zahlen und Zahlworte bekommen auch Ratings und die besten werden ausgewählt. Prinzipiell haben Zahlen Priorität und die Zahlworte richten sich nach ihnen, wenn vorhanden.

5.8 UnitSystem

Ein UnitSystem ist z.B. das Volumen mit den Units Lister, Milliliter etc. Eine Unit kann dabei einen bevorzugten Namen (Singular & Plural) besitzten und weiter Namen/Synonyme nach denen auch gesucht wird. Beim Etrag wird dann der bevorzugte Name ausgegeben.

6. Admin-Bereich
